# Usage: Release Hardening (v0.1)

Status: Draft  
Last Updated: 2026-03-07

This document explains release-hardening checkpoints for `v0.1` users and contributors.
The normative contract is defined in `docs/specs/14_ReleaseHardening_v0.1.md`.

## 1. v0.1 Limitations and Non-Goals

`v0.1` intentionally keeps scope narrow.

- Supported runtime backends: `location: "memory"` and `location: "file"`.
- `location: "browser"` runtime backend is not implemented yet.
- Browser bundle optional profiles (`core-indexeddb`, `core-opfs`, `core-localstorage`, `full-browser`) can remain `planned`.
- Public mutation APIs (`getById`, `updateById`, `deleteById`) are out of scope.
- Registry publishing automation is out of scope.

Post-v0.1 direction is tracked in ADR-51 and prioritizes browser runtime backend implementation.

## 2. Benchmark Method (v0.1 baseline)

Reproducible command:

```bash
pnpm build
pnpm benchmark:v0.1
```

Benchmark script: `scripts/benchmark-v0.1.mjs`

Deterministic dataset shapes:

| Dataset | Backend | Records | Payload shape |
| :------ | :------ | ------: | :------------ |
| `tiny-memory` | `memory` | 1000 | narrow (`event`, `value`, `source`) |
| `small-file` | `file` | 2000 | nested object (depth up to 3) |
| `medium-memory` | `memory` | 10000 | mixed scalar + nested flags |

Reported metrics:

- `insertDurationMs`
- `selectDurationMs`
- `insertRecordsPerSecond`
- `selectRecordsPerSecond`
- environment metadata (`node`, `platform`, `arch`, `cpuModel`)

Interpretation policy:

- local runs are evidence, not strict deterministic performance guarantees
- release review compares current run against previous baseline
- if throughput regresses by more than 20% in any dataset, release notes must include rationale and mitigation plan
