# Spec: Record Format (v0.2 draft)

Status: Draft  
Version: 0.2  
Last Updated: 2026-03-06

This document defines the canonical record model for one timeseries record.
Binary serialization details are defined in `docs/specs/02_BinaryEncoding.md`.

## 1. Scope

- v0.2 supports a mandatory timestamp, an immutable internal insertion-order key,
  and a nested JSON-like payload object.
- payload arrays are still out of scope in v0.2.
- This spec defines the canonical normalized shape. User-facing timestamp input forms are defined in `docs/specs/04_DatastoreAPI.md`.

## 2. Canonical Type Definitions

```typescript
export type SupportedValue = string | number | boolean | null;

export type SupportedNestedValue = SupportedValue | Record<string, SupportedNestedValue>;

export type RecordPayload = Record<string, SupportedNestedValue>;

export interface TimeseriesRecord {
  /**
   * Unix epoch milliseconds.
   * Must satisfy Number.isSafeInteger(timestamp).
   */
  timestamp: number;
  payload: RecordPayload;
}

export interface PersistedTimeseriesRecord extends TimeseriesRecord {
  /**
   * Immutable internal tie-break key.
   * Must satisfy: 0 <= insertionOrder <= 2^64 - 1.
   * This field is generated by datastore and not provided by users.
   */
  insertionOrder: bigint;
}
```

Notes:

- `TimeseriesRecord` is the user-visible logical row shape.
- `PersistedTimeseriesRecord` is the internal persisted shape used by leaf storage and index routing.

## 3. Validation Rules (Normative)

The API boundary (`insert`) MUST normalize user input and enforce all rules below for the canonical record.

### 3.1 Timestamp

- `timestamp` MUST exist.
- `timestamp` MUST be a JavaScript `number`.
- `timestamp` MUST be a safe integer (`Number.isSafeInteger`).
- `timestamp` represents Unix epoch milliseconds.
- Negative timestamps (pre-1970) are allowed.
- Canonical in-memory timestamp domain is intentionally limited to JavaScript safe integers, even though binary storage uses signed `Int64`.
- Decoding from binary MUST reject timestamps outside JavaScript safe integer range; implementations MUST NOT round or clamp.

### 3.2 Payload Object

- `payload` MUST exist and MUST be a non-null plain object.
- Payload values MUST be one of:
  - `string | number | boolean | null`
  - nested plain object (`Record<string, SupportedNestedValue>`)
- Array values MUST be rejected in v0.2.
- Number values in payload (including nested levels) MUST be finite (`Number.isFinite`).
- Circular references in payload object graph MUST be rejected.

### 3.3 Insertion-Order Key (Internal, Persisted)

- Every persisted record MUST include immutable `insertionOrder`.
- `insertionOrder` MUST be generated exactly once at first successful insert.
- `insertionOrder` MUST be `bigint` in memory and within unsigned 64-bit domain
  (`0 <= insertionOrder <= 18446744073709551615n`).
- `insert` input from users MUST NOT accept `insertionOrder`.
- Any mutation operation on an existing record MUST preserve original `insertionOrder`.
- Decoding a persisted record without valid `insertionOrder` MUST fail with typed format/storage error.

### 3.4 Payload Keys

- Keys MUST be non-empty strings.
- Keys are case-sensitive.
- Duplicate keys are not possible in JavaScript objects and are therefore out of scope.
- Key rules apply at every nested object level.

## 4. Ordering Semantics

- Duplicate timestamps are allowed.
- Query result ordering is deterministic:
  - primary key: `timestamp` ascending
  - tie-breaker: `insertionOrder` ascending
- Implementations MUST model tie-break state as an immutable per-record insertion-order key
  assigned exactly once when the record is first inserted.
- The tie-break key MUST be persisted in every leaf record so ordering survives restart,
  compaction/rewrite, and B+ tree split/merge/rebalance.
- Any mutation operation targeting an existing record (for example `updateById`,
  or the update path of a future upsert API) MUST preserve that record's original
  insertion-order key.
- If a future upsert operation takes an insert path (record does not already exist),
  it MUST assign a new insertion-order key exactly as `insert` would.
- Physical storage maintenance (page compaction, page rewrite, B+ tree split/merge/rebalance)
  MUST preserve logical ordering by `(timestamp, insertion-order key)`.

## 5. Error Contract

- Validation failures MUST throw `ValidationError` (see `docs/specs/04_DatastoreAPI.md`).

## 6. Post-Baseline Record Identity Requirement

To support precise update/delete targeting, the canonical record shape is extended after the current baseline:

```typescript
export interface IdentifiedTimeseriesRecord extends TimeseriesRecord {
  /**
   * Immutable internal record identifier.
   * Generated by datastore, not provided by user on insert.
   */
  id: string;
}
```

Requirements:

- every persisted record MUST have one immutable `id`.
- `id` uniqueness MUST hold within a datastore.
- query results that expose identified records MUST preserve deterministic ordering
  rules defined in section 4.
