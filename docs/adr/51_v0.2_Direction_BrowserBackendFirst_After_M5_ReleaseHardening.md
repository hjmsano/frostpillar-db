# ADR-51: v0.2 Direction - Browser Backend First After M5 Release Hardening

Status: Accepted  
Date: 2026-03-07

## Context

All currently defined phase work items (Phase 0-4 closures and supporting M2/M3/P3/P4 slices)
are complete, and active execution is moving to M5 release hardening for `v0.1`.

After M5, Frostpillar needs one explicit `v0.2` direction to avoid parallel scope expansion
between:

1. browser backend runtime support expansion
2. native mutation API expansion (`getById` / `updateById` / `deleteById`)

## Decision

Adopt the following sequencing decision:

- keep active near-term scope on M5 release hardening until exit criteria are met
- set explicit v0.2 direction to browser backend runtime support first
- start with indexedDB/opfs/localStorage runtime-slice implementation and verification
  before adding new public mutation APIs

Rationale:

- current delivery docs already publish browser profile policy and planned profile matrix entries
- browser runtime support is the highest-impact gap between delivery artifacts and runtime capability
- this keeps v0.1 API stability intact while expanding runtime coverage in a bounded way

## Alternatives Considered

1. Prioritize native mutation APIs first after M5.
2. Implement browser runtime and mutation APIs in parallel.

## Consequences

Positive:

- release-to-runtime alignment improves by converting planned browser profiles into test-proven runtime support
- scope remains sequential and reviewable under spec-first/TDD-first governance
- risk of API-surface churn before v0.1 stabilization is reduced

Trade-off:

- mutation API ergonomics are deferred until browser backend baseline is complete
- browser runtime implementation complexity (quota behavior, secure-context constraints) arrives earlier
